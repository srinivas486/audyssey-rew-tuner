1. System Overview

The A1 Evo AcoustiX system bridges Room EQ Wizard (REW) — a professional room measurement tool — with the Denon X3800H AVR's built-in MultEQ XT32 calibration system. The Node.js server acts as a central hub, coordinating measurement data, EQ filter optimization, and AVR telemetry.

  • Browser ↔ Server: HTTP JSON on port 18443 (default; auto-assigned if busy)

Key Insight

The Node.js server is the "brain" — it never stores measurement data long-term. All measurement files (.txt from REW, .ady from Denon MultEQ Editor) are loaded into the browser, which sends them to REW via the server proxy. The server's jobs are: route API calls, manage the Telnet connection to the AVR, and persist calibration files.

2. Measurement Collection Flow

2a. How REW Measures Room Response

REW uses a swept-sine log chirp method to measure the room's frequency response:

  1. REW generates a logarithmic chirp signal (a sine wave that sweeps from ~20 Hz to ~20 kHz, increasing in frequency exponentially)
  2. This chirp is played through the speaker being measured
  3. A calibrated measurement microphone captures the result
  4. REW performs a Fourier transform (FFT) on the recorded signal to extract the impulse response
  5. From the impulse response, the magnitude frequency response is computed
Why Log Chirp?

Logarithmic chirps provide a higher signal-to-noise ratio than sweep-sine or pink noise methods, especially at low frequencies. The logarithmic sweep also provides uniform energy distribution across all octaves, ensuring good measurement accuracy from 20 Hz to 20 kHz in a single sweep.

2b. REW Measurement File Formats

After measurement, REW stores response data in two relevant formats:

.txt (Frequency/Magnitude Export)

Plain text, two columns (frequency in Hz, magnitude in dB). This is the primary format used for importing into REW's EQ module:

[Header lines starting with # are comments] 20.00 -3.2 20.50 -2.9 21.00 -3.1 ... 1000.00 0.5 ... 20000.00 -4.8

.frd (Frequency Response Data)

Same two-column format as .txt (frequency + dB), but without header comments. Used by some external tools.

2c. .ady Files from Denon MultEQ Editor

The Denon MultEQ Editor app (available on iOS/Android) can export calibration sessions as .ady files. These are JSON files containing the full measurement data and detected PEQ filters from Denon's own Audyssey calibration.

The user imports a .ady file via the browser's file input:

Browser File Input → FileReader.readAsText() → parseAdyFile() → normalized channel format

2d. ADY JSON Format

The .ady file is a JSON object with a detectedChannels array. Each channel entry contains the measured response data and the PEQ filters that Audyssey detected:

{ "version": "1.0", "appVersion": "3.0", "detectedChannels": [ { "channelName": "FL", "responseData": [ [20.0, -3.2], [21.0, -2.9], [22.0, -3.1], ... ], "peqFilters": [ { "freq": 63.5, "gain": -8.2, "q": 1.25 }, { "freq": 125.0, "gain": 4.1, "q": 2.0 }, { "freq": 250.0, "gain": -2.7, "q": 3.5 } ] }, { "channelName": "SW", "responseData": [...], "peqFilters": [...] } ], "micCalibration": { "micGain": 0.85 }, "channelReport": { "crossover": 80 } }

2e. parseAdyFile() — Normalization

The parseAdyFile() function in a1evo-calibration.js transforms the Denon format into an internal normalized channel format:

function parseAdyFile(jsonData) {
  // Parse JSON if passed as a string
  if (typeof jsonData === 'string') {
    jsonData = JSON.parse(jsonData);
  }

  const channels = [];
  if (jsonData.detectedChannels) {
    for (const ch of jsonData.detectedChannels) {
      channels.push({
        name:         ch.channelName,           // "FL", "SW", etc.
        responseData: ch.responseData || [],    // [[freq, dB], ...]
        filters: (ch.peqFilters || []).map(f => ({
          freq: f.freq,   // Hz
          gain: f.gain,   // dB
          q:    f.q,      // Q factor
        })),
      });
    }
  }

  return {
    channels,
    micCalibration: jsonData.micCalibration || {},
    channelReport:  jsonData.channelReport  || {},
  };
}

The resulting normalized format is channel-agnostic and used throughout the app for both display and transfer to REW.

3b. Server-Side REW Proxy

In server.js, the proxyRew() function forwards browser requests to REW's localhost API:

const REW_API = 'http://127.0.0.1:4735';

function proxyRew(req, res, pathSuffix, body) {
  return new Promise((resolve, reject) => {
    const url = `${REW_API}${pathSuffix}`;
    const options = {
      method: body ? 'POST' : 'GET',
      headers: { 'Content-Type': 'application/json' },
    };
    const fetchReq = http.request(url, options, (fetchRes) => {
      let data = '';
      fetchRes.on('data', (chunk) => data += chunk);
      fetchRes.on('end', () => {
        try {
          resolve({ status: fetchRes.statusCode, body: JSON.parse(data) });
        } catch {
          resolve({ status: fetchRes.statusCode, body: data });
        }
      });
    });
    fetchReq.on('error', reject);
    if (body) fetchReq.write(JSON.stringify(body));
    fetchReq.end();
  });
}

The browser calls /api/rew/eq/match-target, and the server forwards it to http://127.0.0.1:4735/eq/match-target.

3c. Client-Side REW API (a1evo-rew.js)

The browser-side library a1evo-rew.js wraps all REW calls with fetch() to the server proxy:

async function rewImportImpulse(channel, irData, sampleRate = 48000) {
  // Convert Float32Array → base64 Big-Endian IEEE 754 float32
  let buffer;
  if (irData instanceof Float32Array) {
    buffer = new ArrayBuffer(irData.length * 4);
    const view = new DataView(buffer);
    for (let i = 0; i < irData.length; i++) {
      view.setFloat32(i * 4, irData[i], false); // Big-Endian
    }
  } else {
    buffer = irData.buffer;
  }
  const base64 = btoa(String.fromCharCode(...new Uint8Array(buffer)));

  const res = await fetch('/api/rew/eq/import-impulse', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ channel, irData: base64, sampleRate }),
  });
  return res.json();
}

async function rewMatchTarget(channel, targetCurveFile, options = {}) {
  const res = await fetch('/api/rew/eq/match-target', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ channel, targetCurve: targetCurveFile, ...options }),
  });
  return res.json();
}

4. Target Curve System

What Is a Target Curve?

A target curve defines the desired frequency response shape — what you want the room + speaker system to sound like after EQ is applied. REW's EQ optimization algorithm generates filters that bring the measured response as close as possible to the target curve.

Target curves typically:

  • Roll off the very low frequencies slightly (e.g., -3 dB at 20 Hz) to account for room gain and protect subwoofers
  • Stay flat through the midrange (the "neutral" region)
  • Apply a gentle high-frequency roll-off (mimicking the natural sound of a well-treated room)

File Location and Format

Target curves live in target_curves/ next to server.js:

Note on Frequency Units

REW uses decihertz (dHz) in its target curve files — i.e., 1000 Hz is written as 1000, not 100.0. The default AcoustiX curve uses this format. If you're creating custom curves, make sure frequencies are in dHz (whole Hz units).

5. Filter Generation Algorithm

How REW's EQ Module Works

REW's EQ module is a parametric equalizer (PEQ) optimizer. Given a measured frequency response and a target curve, it automatically:

  1. Detects peaks in the measured response using an automated peak-detection algorithm
  2. Creates matching inverse filters — each PEQ filter has the opposite magnitude response of the peak it targets
  3. Runs an optimization loop that adjusts filter frequency, gain, and Q to minimize the error between the measured response and the target curve
  4. Respects limit constraints (max cut, max boost, frequency range, number of filters)

Parametric EQ Filter Parameters

Each PEQ filter is defined by three parameters:

ParameterDescriptionTypical Range
Frequency (Hz)Center frequency of the filter20 – 20,000 Hz
Gain (dB)Amount of cut or boost at the center frequency. Negative = cut, positive = boost.–15 to +6 dB
QQuality factor — bandwidth of the filter. Higher Q = narrower filter.0.5 – 20

Filter Count Limits

  • Speakers (FL, FR, C, etc.): up to 6 PEQ filters per channel
  • Subwoofer (SW): up to 7 filters (6 + 1 additional for low-frequency correction)
  • REW may generate fewer filters than the maximum if the response doesn't require them

Target Curve Matching Process

The full optimization sequence per channel:

1. rewImportImpulse(channel, irData, sampleRate) → REW loads the impulse response for this channel 2. rewMatchTarget(channel, targetCurveFile, options) → REW runs: detect peaks → create inverse filters → optimize → Returns: { filters: [{freq, gain, q}, ...], targetMatch: %, deviation: dB } 3. rewExportRaw(channel) → Optionally export biquad coefficients for external use

Limit Parameters

When calling rewMatchTarget(), the following constraints can be passed:

OptionDefaultDescription
maxFilters6Maximum number of PEQ filters to generate
maxCut12 dBLargest cut allowed per filter
maxBoost6 dBLargest boost allowed per filter
freqLimitLow20 HzLowest frequency for EQ correction
freqLimitHigh20000 HzHighest frequency for EQ correction
targetLevel0 dBReference level for matching

Subwoofer vs Speaker Handling

Subwoofers are handled differently because their frequency range is limited and room effects are more severe at low frequencies:

  • Subwoofers get an additional filter slot (7 vs 6) for deep-bass corrections
  • The target curve for subwoofers typically starts with a steeper low-frequency roll-off
  • Crossover settings (e.g., 80 Hz) are managed by the AVR independently, not by REW

6. AVR Telemetry Protocol (Telnet Port 23)

Overview

The Denon X3800H exposes a command-line interface over raw TCP Telnet (port 23). The Node.js server maintains a persistent Telnet connection to the AVR and sends commands to configure PEQ filters, distances, and trims. This is the same protocol used by Denon's own apps and calibration software.

Connection State Machine

The AVR connection follows a strict state machine: disconnected → connecting → ready. Commands are only sent when ready === true. The server maintains a single in-memory socket singleton.

Response Format

All responses from the AVR are terminated with \r (carriage return). The server accumulates data until \r is detected, then resolves the promise:

function avrSendCommand(cmd) {
  return new Promise((resolve, reject) => {
    let response = '';
    const timeout = setTimeout(() => {
      avrSocket.removeListener('data', onData);
      reject(new Error('Command timeout'));
    }, 5000);   // ← 5 second timeout per command

    function onData(data) {
      response += data.toString();
      if (response.includes('\r')) {
        clearTimeout(timeout);
        avrSocket.removeListener('data', onData);
        resolve(response.trim());
      }
    }

    avrSocket.on('data', onData);
    avrSocket.write(cmd + '\r', (err) => {
      if (err) { clearTimeout(timeout); reject(err); }
    });
  });
}

AvrClient Class (avr/avr-client.js)

The full AvrClient class wraps all Telnet operations with queuing, timeout handling, and connection state management:

class AvrClient {
  constructor() {
    this.socket = null;
    this.host = null;
    this.ready = false;
    this._commandQueue = [];
    this._processingQueue = false;
    this._responseHandlers = new Map();
    this._buffer = '';
  }

  async connect(host) {
    if (this.socket && !this.socket.destroyed) this.socket.destroy();
    this.host = host;
    this.socket = net.connect(23, host);

    return new Promise((resolve, reject) => {
      const timeout = setTimeout(() => {
        this.socket.destroy();
        reject(new Error('Connection timeout (5s)'));
      }, 5000);

      this.socket.on('connect', () => {
        clearTimeout(timeout);
        this._onceData((data) => {
          this.ready = true;
          resolve({ host, port: 23, connected: true });
        });
        this.socket.write('?\r');
      });

      this.socket.on('error', reject);
    });
  }

  async sendCommand(cmd) {
    return new Promise((resolve, reject) => {
      const timer = setTimeout(() => {
        this._responseHandlers.delete('line');
        reject(new Error(`Command timeout: ${cmd}`));
      }, 5000);

      this._responseHandlers.set('line', (line) => {
        clearTimeout(timer);
        this._responseHandlers.delete('line');
        resolve(line);
      });

      this.socket.write(cmd + '\r', (err) => {
        if (err) { clearTimeout(timer); reject(err); }
      });
    });
  }
}

7. SSDP AVR Discovery (UDP Multicast 239.255.255.250:1900)

How It Works

Denon/Marantz AVRs implement SSDP (Simple Service Discovery Protocol) for zero-configuration network discovery. The server sends an M-SEARCH UDP multicast request to the reserved SSDP address 239.255.255.250 on port 1900. Any compatible AVR on the local network responds with its device description URL.

M-SEARCH Request Format

M-SEARCH * HTTP/1.1
HOST: 239.255.255.250:1900
MAN: "ssdp:discover"
MX: 3
ST: urn:schemas-denon-com:device:ACT-DenonDeviceService:1

Key headers:

  • MAN: SSDP discover protocol identifier
  • MX: Maximum wait time (seconds) — the AVR waits 0–3 seconds before responding
  • ST: Search target — urn:schemas-denon-com:device:ACT-DenonDeviceService:1 specifically targets Denon/Marantz devices

Parsing Responses

AVR responses include a LOCATION header containing the device's XML description URL (which contains the AVR's IP address):

NOTIFY * HTTP/1.1
LOCATION: http://192.168.1.100:8080/description.xml
SERVER: Linux/2.6 UPnP/1.0 Denon/1.0
USN: uuid:Denon-ACT-DenonDeviceService:1
NT: urn:schemas-denon-com:device:ACT-DenonDeviceService:1

The LOCATION URL's hostname is the AVR's IP address. The server parses this and returns it to the browser for Telnet connection.

ssdpDiscover() Implementation

function ssdpDiscover() {
  return new Promise((resolve, reject) => {
    const client = dgram.createSocket({ type: 'udp4', reuseAddr: true });
    const timeout = setTimeout(() => {
      client.close();
      resolve([]);   // Return empty on timeout (not an error)
    }, 4000);

    const results = [];

    client.on('message', (msg, rinfo) => {
      const str = msg.toString();
      // Filter for Denon/Marantz devices
      if (str.includes('Denon') || str.includes('Marantz') ||
          str.includes('ACT-DenonDevice')) {
        const locationMatch = str.match(/LOCATION:\s*(.+)/i);
        results.push({
          ip:       rinfo.address,
          location: locationMatch ? locationMatch[1].trim() : null,
          server:   str.match(/SERVER:\s*(.+)/i)?.[1].trim(),
          usn:      str.match(/USN:\s*(.+)/i)?.[1].trim(),
        });
      }
    });

    client.bind(() => {
      client.setBroadcast(true);
      const search = [
        'M-SEARCH * HTTP/1.1',
        'HOST: 239.255.255.250:1900',
        'MAN: "ssdp:discover"',
        'MX: 3',
        'ST: urn:schemas-denon-com:device:ACT-DenonDeviceService:1',
        '', ''
      ].join('\r\n');

      client.send(search, 0, search.length, 1900, '239.255.255.250');
    });

    client.on('error', (err) => {
      clearTimeout(timeout);
      client.close();
      reject(err);
    });
  });
}
Timeout Is Normal

The 4-second timeout is intentional — after sending the M-SEARCH, the function waits up to 3 seconds (matching MX: 3) for AVRs to respond, then returns whatever it found. An empty array means no Denon/Marantz devices were found on the network.

8. Calibration File Formats

.oca Format (A1 Evo Native)

.oca is A1 Evo AcoustiX's own calibration format — a JSON file that stores the complete calibration state: AVR info, per-channel PEQ filters, distances, trims, target curve name, and all form settings. This is the file format used for save/load in the app.

1
SSDP AVR Discovery
User clicks "Discover AVR". Browser calls POST /api/avr/discover. Server sends M-SEARCH UDP multicast. AVRs respond with LOCATION headers containing their IPs.
ssdpDiscover() → [{ip: "192.168.1.100", ...}, ...]
2
Connect via Telnet
User selects an AVR IP and clicks "Connect". Server opens raw TCP socket to AVR port 23. AVR sends telnet negotiation bytes. Connection state → ready.
POST /api/avr/connect {host: "192.168.1.100"} → {success: true}
3
Load Measurement
User loads a measurement from REW (.txt file export) or Denon MultEQ Editor (.ady file). The browser parses the file and stores the response data for each channel. Alternatively, REW is used to make live measurements.
parseAdyFile(adyJson) → {channels: [{name, responseData, filters}]}
4
Select Target Curve
User picks a target curve from the dropdown (e.g., "acoustix.txt"). The curve file is loaded from GET /api/target-curves/:name. The curve defines the desired frequency response shape.
GET /api/target-curves/acoustix.txt → "20\t-3.0\n1000\t0.0\n..."
5
REW Optimizes Filters
For each channel: (a) Import impulse response via /api/rew/eq/import-impulse. (b) Run target matching via /api/rew/eq/match-target. REW returns PEQ filter parameters (freq, gain, Q). (c) Optionally export coefficients via /api/rew/eq/export-raw.
rewMatchTarget("FL", "acoustix.txt") → {filters: [{freq:63.5,gain:-8.2,q:1.25},...]}
6
Transfer Filters to AVR
Server sends PEQ commands to the AVR via Telnet, one filter at a time with small delays between. Each command follows the MSSV format.
MSSVFL=63.5Hz,-8.2dB,Q=1.25\r
7
Set Speaker Distances
Server sends MSD commands for each channel, encoding the distance in millimeters.
MSDFL3450\r (3450 mm = 3.45 m)
8
Set Channel Trims
Server sends MST commands for each channel, encoding trim in 0.1 dB units.
MSTFL105\r (+10.5 dB)
9
Save Calibration as .oca
Browser serializes the complete state (AVR info, channels, filters, distances, trims, settings) into an .oca JSON file and POSTs it to the server. Server writes it to calibrations/<name>.oca.
POST /api/calibration {name: "Movie Night", data: {...}}

10. Server API Routes

The Node.js Express-style router in server.js handles all HTTP API calls. The server is a custom HTTP server (not Express) but follows Express routing patterns.

RouteMethodHandlerDescription
/api/avr/discover POST ssdpDiscover() Send SSDP M-SEARCH multicast. Returns array of discovered AVRs with IP, LOCATION, SERVER, USN.
/api/avr/connect POST avrConnect(body.host) Open raw TCP Telnet connection to AVR at host:23. 5s timeout. Returns {success: true, host, port}.
/api/avr/disconnect POST avrDisconnect() Destroy the Telnet socket. Returns {disconnected: true}.
/api/avr/command POST avrSendCommand(body.command) Send raw command to AVR over Telnet. 5s timeout. Returns {success: true, response}.
/api/avr/query POST avrSendCommand('MSQ' + body.channel) Query PEQ filter state for a channel. Body: {channel: "FL"}.
/api/rew/* GET / POST proxyRew(req, suffix, body) Proxy all requests to REW API at localhost:4735. Strips /api/rew prefix before forwarding.
/api/calibration GET listCalibrations() List all .oca files in calibrations/. Returns {success: true, calibrations: [{name, file}, ...]}.
/api/calibration POST saveCalibration(body.name, body.data) Write calibration to calibrations/<name>.oca. Body: {name, data: ocaObject}.
/api/calibration/:name GET loadCalibration(name) Read and return calibrations/<name>.oca. Returns full OCA JSON object.
/api/target-curves GET listTargetCurves() List all curve files in target_curves/. Returns {success: true, curves: [{name, file}, ...]}.
/api/target-curves/:name GET getTargetCurve(name) Return raw text content of the curve file (frequency/gain pairs).
/api/parse-ady POST JSON parse body.adyData Parse and validate a Denon .ady JSON file. Returns {success: true, data: parsed} or error.

Static File Serving

Any request that doesn't match an API route is served as a static file from the server's directory. This includes the HTML, CSS, and JavaScript for the SPA:

  • / or /index.html → index.html
  • /css/a1evo.css → css/a1evo.css
  • /js/a1evo-app.js → js/a1evo-app.js
  • /calibrations/myname.oca → downloadable file

11. Baud Rates and Limitations

MultEQ XT32 Communication Speed

The Denon X3800H's Telnet interface communicates at 19200 baud (19.2 kbps). This is a relatively slow serial speed by modern standards, which has direct implications:

  • Each command string is small (~30–50 bytes), but the AVR's response processing adds latency
  • Sending many PEQ commands in rapid succession can cause the AVR to drop or corrupt responses
  • The 50ms inter-command delay in setPEQBatch() is specifically chosen to avoid overflow

PEQ Filter Limits

Channel TypeMax PEQ FiltersNotes
Regular speakers (FL, FR, C, SL, SR, etc.)6Per channel
Subwoofer (SW)76 + 1 extra for deep bass
Height channels (FD, FS, TM, RL, RR)6Same as regular

Filter Transfer Strategy

Because of the 19200 baud limit and 5-second command timeout, the app transfers filters in batches with inter-command delays:

async function setPEQBatch(channel, filters) {
  const results = [];
  for (const f of filters) {
    const r = await setPEQ(channel, f.freq, f.gain, f.q);
    results.push(r);
    // 50ms delay prevents AVR command buffer overflow
    await new Promise(r => setTimeout(r, 50));
  }
  return results;
}
Why Small Batches with Delays?

If you send commands too fast, the AVR's 19200-baud input buffer overflows and the command is truncated or ignored. The 5-second timeout catches this case — the command hangs and eventually times out, leaving the filter in an unknown state. The 50ms delay between commands avoids this reliably.

Other Known Limitations

  • Single Telnet connection: The server maintains one in-memory socket. If the connection drops mid-transfer, the user must reconnect manually.
  • REW must be running: The REW API (port 4735) only responds when REW is actively running on the same machine. There's no wake-on-LAN for REW.
  • Subwoofer crossover: The crossover frequency (e.g., 80 Hz) is set by the AVR independently and is not managed by this app.
  • Port 18443: If this port is busy, the server auto-increments to 18444, 18445, etc. Check the console output for the actual URL.
  • No HTTPS: The server runs over plain HTTP on a local network. This is fine for local use but should not be exposed to the internet.
